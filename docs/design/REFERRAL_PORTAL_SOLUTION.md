**Дата:** 2026-06-22

**Рішення продукту:** повна крос-клінічна модель 2.0 (глобальний направник, M2M-доступ до центрів)

## 1. Що це за документ

Це не новий дизайн, а **зафіксоване оптимальне рішення** поверх дизайн-документа: те саме бачення (глобальний крос-клінічний направник), але з виправленням трьох неточностей дизайну, які інакше зламали б реалізацію, і з готовим, перевіреним на реальній схемі бекендом. Дизайн-документ лишається довідником «чому»; цей файл — «що саме робимо й у якому порядку».

## 2. Висновок аналізу попередньої роботи

Дизайн-документ архітектурно правильний: зберегти `queue_entries.clinic_id`/`room_id` і тригери `check_no_overlap`/`one_in_progress`/`no_booking_during_incident` означає, що **безконфліктність черг працює для крос-клінічних записів безкоштовно** — жодного нового механізму блокування не потрібно. Це головна сильна сторона, яку лишаємо без змін.

Під час звірки з кодом і схемою знайдено **три розбіжності**, які увійшли в оптимальне рішення:

| # | Що в дизайні | Реальність / ризик | Рішення |
|---|---|---|---|
| **A** | `queue_select` описано двічі — спершу версія з `clinic_id IN auth_referrer_clinics()` (віддає направнику сирі чужі `queue_entries` → **витік ПІБ/телефонів** між центрами), потім виправлена | Перша версія — реальний витік PII | Зафіксовано **лише** виправлену: направник читає в `queue_entries` **тільки власні** (`created_by = auth.uid()`); чужа зайнятість — через знеособлений RPC |
| **B** | «потрібно додати `clinics.city` через ALTER» | `clinics.city` **вже існує** з міграції 0002 | ALTER прибрано; `search_clinics` одразу використовує `city` |
| **C** | *(не згадано)* | Коли `clinic_id` стає NULL, `auth_clinic_id()` повертає NULL, і наявна `profiles_select` (`clinic_id = auth_clinic_id()`) **блокує направнику читання власного профілю** → `/referral` падає на завантаженні | Додано self-політику `profiles_select_self (id = auth.uid())` у 0023 |

Пункт **C** — критичний: без нього вся модель не злетіла б, бо сторінка `app/referral/page.tsx` першим запитом читає власний `profiles`-рядок.

## 3. Зафіксована модель (коротко)

- **`referral_access`** (M2M направник↔центр) — єдине джерело істини про членство. Статуси: `pending_clinic`, `pending_referrer`, `active`, `revoked`, `declined`.
- **`profiles.clinic_id` → nullable.** NULL ⇔ глобальний направник; NOT NULL ⇔ персонал центру.
- **Хелпери:** `auth_referrer_clinics()` (множина active-центрів), `auth_can_refer(c)` (булеан).
- **RLS:** персонал — своя клініка; направник **читає** `rooms`/`incidents`/`schedule_overrides`/`clinics` авторизованих центрів і **пише** власні `queue_entries` у будь-який авторизований центр; у `queue_entries` **читає лише власні**.
- **RPC:** `room_busy_slots(room, date)` — знеособлені інтервали зайнятості; `search_clinics(q)` — каталог налаштованих центрів.
- **Безконфліктність:** наявні серверні тригери, без змін.
- **Realtime:** один канал `filter: created_by=eq.<id>` на всі центри + канал на обраний кабінет у формі.

## 4. План впровадження

### Етап A — Дані й доступ ✅ реалізовано (міграції, запускати по черзі в Supabase)
- **0023_referrer_global.sql** — `referral_access` + enum; `clinic_id` nullable; `auth_referrer_clinics()`, `auth_can_refer()`; **`profiles_select_self`**; RLS на `referral_access`.
- **0024_referrer_rls.sql** — політики `rooms` (staff/referrer), `queue_entries` (select own-only, write через `auth_can_refer`), `incidents`, `schedule_overrides`, `clinics`.
- **0025_referrer_rpc.sql** — `room_busy_slots`, `search_clinics` (без ALTER city).
- **0026_migrate_existing_referrers.sql** — бек-філ `referral_access(active)` + обнулення `clinic_id`. Ідемпотентна.

> ⚠️ БД спільна з продакшеном. Запускати **по порядку**, по одній, у SQL Editor. Після 0026 перевірити: `select count(*) from profiles where role='referrer' and clinic_id is not null;` має бути `0`.

### Етап B — Серверні роути (service_role, наступний крок)
- `POST /api/referral/access/request` — направник надсилає запит у центр (`pending_clinic`); дедуплікація через `unique(referrer_id, clinic_id)`.
- `POST /api/referral/access/decide` — підтвердження/відхилення; перевірка, що рішення приймає правильна сторона (адмін центру для `pending_clinic`; направник для `pending_referrer`).
- `POST /api/referrers/invite` — центр запрошує направника за email: якщо акаунт є → `referral_access(pending_referrer)`; якщо нема → створити **глобальний** referrer-акаунт (`clinic_id = NULL`) + `pending_referrer` (інакше, ніж поточний `/api/staff`, який садить у клініку).

### Етап C — UI
- `app/referral/page.tsx` — прибрати єдиний `clinicId`/`rooms`; вантажити `referral_access` (active+pending) і дані центрів; передавати список центрів.
- `ReferralPortal.jsx` — вкладка **«Мої центри»**; вибір центру в «Новому направленні» (центр → кабінет → слоти через `room_busy_slots`); колонка «Центр» у «Моїх направленнях»; **кнопка «Скасувати»** (зараз її немає); realtime по `created_by` + по обраному кабінету.
- `ReferrersManager.jsx` — вхідні запити направників (підтвердити/відхилити), запрошення направника, відкликання доступу.

### Етап D — Верифікація
RLS-тести (направник A не бачить PII пацієнтів направника B; не бачить неавторизовані центри; не пише в неавторизований центр); конкурентний тест слота (два направники + адмін → один успіх); realtime крізь різні браузери; перевірка, що `room_busy_slots` не повертає ПІБ/телефон.

## 5. Ризики (доповнення до §9 дизайну)

- **`clinic_id` nullable б'є по припущеннях NOT NULL.** Аудит зроблено для критичного шляху (`profiles_select` → додано self-політику). Перед Етапом C перевірити решту місць, що читають `profile.clinic_id` для направника, і направити їх на `auth_referrer_clinics()`/`referral_access`.
- **Інвайт-шлях `handle_new_user` (0009).** Якщо направника запросити через наявний `clinic_invites`, він стане однотенантним (`clinic_id = inv.clinic_id`). Тому в Етапі B запрошення направника йде окремим роутом, що створює глобальний акаунт, а не через `clinic_invites`.
- **Спільна prod-БД.** Усі міграції ідемпотентні й не валідують наявні рядки; розгортання поетапне, бекенд невидимий для UI до Етапу C.

## 6. Відкриті питання (для наступного раунду)
Майстер-реєстр пацієнтів (дедуплікація між центрами); сповіщення направнику (email/Viber/Telegram); «Порівняти центри» (найближчі вільні слоти поряд); звіти направника; політика видимості `indication`; передача висновку (`PDF`) направнику.

## 7. Звірка з product-спекою v4.0/v5.0 (з вхідного матеріалу)

Вхідний продуктовий розбір (лінія спеки v4.0/v5.0) підтверджує архітектуру цього рішення і додає кілька корисних ідей. Важливо: той розбір написаний у **термінах спеки v4.0** (`appointments`, `machine_id`, дискретні слоти `UNIQUE(machine_id, slot_start)`, синхронність через **SWR-полінг ~10 с**, PostgREST), тоді як **реалізований** RadFlow інший: `queue_entries`+`created_by`+`room_id`, вільний `scheduled_time`(text)+`duration_min` під тригером `check_no_overlap` (advisory-lock), синхронність через **Supabase realtime**. Тому частину рекомендацій транслюємо, частину лишаємо як є.

### 7.1. Що збігається (підтверджує наше рішення)
- **M2M-доступ** `referrer_access` ≈ наша `referral_access` (єдине джерело істини про членство).
- **Глобальний направник над тенантами.** У спеці — окрема сутність `referrer` без `clinic_id`; у нас те саме досягнуто через `profiles.role='referrer'` + `clinic_id IS NULL` (щоб не форкати identity/auth.users — нижчий ризик, той самий результат M2M).
- **Двовимірна RLS:** доступність центру (через грант) + власність направлення (`referred_by` ≈ наш `created_by`). Реалізовано в 0024.
- **Безконфліктність на рівні БД, а не UI.** Збігається з нашим висновком: гарантію дає сервер. У спеці — `UNIQUE(machine_id, slot_start)`; у нас — `check_no_overlap` (advisory-lock per room по діапазону часу). Обидва дають залізну гарантію; наш уже працює.
- **Фінальна влада оператора/clinic admin** над своїм кабінетом; зміни — з повідомленням, не мовчки.

### 7.2. Що адаптуємо/перекладаємо
- **Синхронність:** лишаємо **Supabase realtime** (краще за полінг 10 с зі спеки) + полінг-підстрахування на `focus/visibilitychange`. Не відкочуємось на SWR-полінг.
- **Атомарне бронювання:** ідея спеки про RPC `book_slot`/`move_appointment` слушна. У нас вставку вже робить атомарно-безпечною тригер; для confirm-режиму та soft-hold обгорнемо в Supabase RPC (security definer) на Етапі B.
- **Дискретні слоти:** не вводимо — RadFlow свідомо моделює час як `scheduled_time+duration_min`. Сітка 30-хв слотів — лише відображення; гарантію дає тригер.

### 7.3. Що додаємо (нове й цінне зі спеки)
- ✅ **`referral_access.policy` (direct / confirm)** — додано в 0023. `direct` = одразу в чергу; `confirm` = направлення як «заявка», поки оператор не підтвердить. Логіка confirm — Етап B/C.
- **Soft-hold слота** (`slot_holds` + TTL + чистка фоновим job) — UX-покращення «притримати слот, поки заповнюєш форму». Коректність уже гарантує тригер, тож це **опційний** наступний крок, не блокер MVP.
- **`audit_log` + повідомлення направнику** при переносі/скасуванні оператором — закриває «конфлікт повноважень» прозоро. Окремий невеликий епік (з §6 сповіщеннями).
- **Каталог «Моя мережа»** — розширення `search_clinics`: фільтри місто/радіус, тип апарата (1.5T/3T, контраст), ціна, найближчі вікна; мапа+список. MVP: простий пошук (вже є RPC); v2 — фільтри/мапа.
- **Статус «Висновок готовий»** направнику (від радіолога) — продуктове рішення про периметр даних (див. §6).

### 7.4. Продуктові рішення, що лишаються за тобою
1. **Гранулярність гранту:** зафіксовано **рівень центру** (попереднє рішення). Спека пропонує опційний `cabinet_id`. Якщо потрібен — додамо nullable `cabinet_id` у `referral_access` (NULL = весь центр), хелпери лишаться клінік-рівневими доки не знадобиться.
2. **Стартова `policy`:** `direct` (швидкий UX) чи `confirm` (контроль кабінету) за замовчуванням. Зараз дефолт схеми — `direct`.
3. **Каталог:** відкритий для пошуку всіма направниками (доступ — лише за підтвердженням центру) чи лише за запрошенням. Зараз `search_clinics` показує всі налаштовані центри (відкритий для пошуку, доступ потребує підтвердження).
4. **Висновок направнику:** віддаємо PDF/висновок чи лише статус «готово».
