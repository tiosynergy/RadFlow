## 1. Резюме

Пять рисков, заявленных к анализу, при проверке по коду сводятся к четырём задачам техдолга с общим корнем (отсутствие типизации Supabase) плюс один реальный операционный риск (realtime + поллинг). Делать строго по порядку: типы → бизнес-логика → realtime → слой данных → компоненты.

| ID | Риск | Приоритет | Оценка | Зависит от |
|----|------|-----------|--------|------------|
| TD-1 | Нет `supabase/types.ts` / клиенты без `<Database>` | 🔴 P1 (рычаг) | 1–2 ЧД | — |
| TD-2 | Бизнес-логика в `.js` (`lib/*.js`) | 🟠 P2 | 2–3 ЧД | TD-1 |
| TD-3 | Realtime: полный refetch + безусловный поллинг 10с | 🔴 P1 (прод) | 3–5 ЧД | — (можно || с TD-1) |
| TD-4 | Нет Server Actions / кэша; чтения и мутации с клиента | 🟠 P2 | 5–8 ЧД | TD-1 |
| TD-5 | Все компоненты в `.jsx` (legacy прототипа) | 🟡 P3 (фон) | 8–13 ЧД | TD-1 |

**Итого:** ~19–31 ЧД. Легенда оценки: ЧД = идеальный человеко-день одного инженера; диапазон = без/с непредвиденными.

## 2. Фактическая картина (на 2026-06-27)

- **Компоненты:** 23 продакшен-файла в `components/` — **все `.jsx`**, `.tsx`-компонентов нет. 13 страниц `app/**/page.tsx` — тонкие, делегируют в `.jsx`. Ещё 8 `.jsx` в `docs/prototypes/code/` — мёртвый код прототипа в дереве.
- **lib/:** инфраструктура типизирована (`auth.ts`, `rateLimit.ts`, `supabase/{client,server,admin,middleware}.ts`); **бизнес-логика — `.js`**: `studies.js`, `incidents.js`, `queueStatus.js`, `schedule.js`.
- **Типы Supabase:** `supabase/types.ts` отсутствует. `createClient()` в `lib/supabase/server.ts` вызывает `createServerClient(...)` **без дженерика `<Database>`** → каждый `.from(...).select(...)` возвращает `any`.
- **Слой данных:** Server Actions (`'use server'`) — **ноль**. Серверная логика есть только в Route Handlers (`app/api/staff`, `app/api/account/set-password`, `app/api/referral/*`, `app/auth/callback`). Доски читают и значительную часть мутаций делают **напрямую с клиента** анон-клиентом (напр. `QueueBoard.jsx:710` — `supabase.from("incidents").update(...)`). React `cache()` / `unstable_cache` / `revalidateTag` / TanStack Query не используются.
- **Realtime:** в `QueueBoard`, `RadiologistBoard`, `CeoDashboard`, `CallListBoard` — канал на клинику с `filter: clinic_id=eq.<id>` (тенант-изоляция корректна), но подписка `event:"*"` на целые таблицы, и обработчик делает **полный `reload()`** + поверх крутится **`setInterval(refetchAll, 10000)`** безусловно.

## 3. Инициативы

### TD-1. Сгенерировать `supabase/types.ts` и типизировать клиентов 🔴 P1
**Файлы:** новый `supabase/types.ts`; `lib/supabase/{server,client,admin,middleware}.ts`.
**Почему первым:** корень риска. Пока доступ к БД — `any`, миграция в TS (TD-2, TD-5) даёт ложную защиту: типобезопасность кончается на границе с Supabase.
**План:**
1. `supabase gen types typescript --project-id <radflow> --schema public > supabase/types.ts` (или `--linked`).
2. В каждом клиенте: `createServerClient<Database>(...)`, `createBrowserClient<Database>(...)`, импорт `Database` из `supabase/types.ts`.
3. Прогнать `tsc --noEmit`, починить вскрывшиеся несоответствия (ожидаемы в `lib/*.js`/`.jsx` — частично закроются в TD-2/TD-5).
4. Добавить шаг генерации в CI и пометку в `CONTRIBUTING`: **после каждой миграции пересоздавать types.ts** (иначе типы разъедутся со схемой).
**Оценка:** 1–2 ЧД. **Риск:** низкий, поведение не меняется.
**Критерий готовности:** `tsc --noEmit` зелёный; `.from("queue_entries")` даёт типизированный результат.

### TD-2. Перевести бизнес-логику `lib/*.js` → `.ts` 🟠 P2
**Файлы:** `lib/studies.js`, `lib/incidents.js`, `lib/queueStatus.js`, `lib/schedule.js`.
**Почему:** бизнес-логика — худшее место для отсутствия типов (целостность данных, тенант-изоляция). Раньше компонентов.
**План:**
1. Начать со `studies.js` — единственный источник истины по составу исследований (`queue_entries.studies` + `studies_original` JSONB, закреплено в аудите MAJ-12).
2. Переименовать в `.ts`, проставить явные сигнатуры и return-типы, опираясь на `Database` из TD-1.
3. Ввести Zod-схемы для парсинга JSONB (`studies`/`studies_original`) на входе — runtime-валидация там, где типы TS не помогают.
4. Устранить попутно MIN-3 аудита (разные определения «пострадавших от инцидента» в `incidents.js`) — единая функция.
**Оценка:** 2–3 ЧД. **Зависит от:** TD-1. **Риск:** средний (можно вскрыть скрытые неявные контракты).

### TD-3. Чинить realtime: инкрементальные апдейты вместо полного refetch + убрать безусловный поллинг 🔴 P1
**Файлы:** `QueueBoard.jsx`, `RadiologistBoard.jsx`, `CeoDashboard.jsx`, `CallListBoard.jsx`.
**Почему острее остальных:** риск уже в проде, не «потенциальный». Усиление нагрузки:
- `event:"*"` + полный `reload()` → одно изменение статуса в большой клинике заставляет **все** подключённые клиенты (очередь + радиолог + CEO + call-list) рефетчить очередь целиком.
- `setInterval(refetchAll, 10000)` работает **всегда**, даже без изменений: базовая нагрузка `клиенты × 3 запроса / 10с`, независимая от realtime.
- Каждая доска держит свой канал + свой поллинг; открытые одновременно очередь+радиолог+CEO = 3 канала + 3 поллинга на клинику.
Итог: нагрузка ≈ `клиенты × (рефетч на каждое изменение + поллинг каждые 10с) × несколько запросов`. Плохо масштабируется по числу кабинетов/клиник.
**План:**
1. Обрабатывать `payload.new`/`payload.old` и обновлять локальный стейт точечно вместо полного `reload()` (фолбэк-`reload()` оставить только на необъяснимых рассинхронах).
2. Заменить безусловный 10-секундный `setInterval` на refetch по `visibilitychange`/`focus` + поллинг с backoff **только** при разрыве realtime-сокета (статус подписки ≠ `SUBSCRIBED`).
3. Свести дублирующиеся подписки одной доски в один канал; сузить колонки/`event` (`UPDATE` вместо `*` там, где INSERT/DELETE не нужны).
4. Для высокочастотных статусов кабинетов оценить переход с `postgres_changes` на **Broadcast** (в т.ч. из триггеров БД) — без проверки RLS на каждое событие.
5. Замерить до/после: число запросов к Supabase в минуту на N открытых клиентов.
**Оценка:** 3–5 ЧД. **Риск:** средний (нужно аккуратно с консистентностью; обязателен сценарный прогон ролей).
**Критерий готовности:** при идле сетевой трафик к БД ≈ 0; одно изменение → один точечный апдейт у подписчиков.

### TD-4. Ввести слой данных: Server Actions + кэш 🟠 P2
**Файлы:** новые `app/**/actions.ts`; рефактор мутаций из досок; чтения в `page.tsx`.
**Почему:** сейчас мутации в клиентских `.jsx` нарушают правила проекта (Server Components + Server Actions + optimistic updates) и держат безопасность только на RLS, без серверной валидации. «Много прямых вызовов» = отсюда.
**План:**
1. Вынести записи (`update incidents`, `saveOverride`, вызовы в кабинет и т.п.) в Server Actions с `revalidateTag`/`revalidatePath`; в компонентах — `useOptimistic`/`useTransition`.
2. Чтения для первичного рендера перенести в Server Components; дедуп в рамках запроса — React `cache()`; межзапросный кэш — `unstable_cache` с тегами.
3. **TanStack Query — только** для realtime-досок (живые клиентские вью), не для RSC-страниц (там он дублирует кэш Next).
4. Единый error-handling и loading-состояния на уровне actions (правила проекта).
**Оценка:** 5–8 ЧД (поэтапно, по модулям). **Зависит от:** TD-1. **Риск:** средний-высокий — меняет поток данных; катить по одному модулю с регрессом.

### TD-5. Мигрировать компоненты `.jsx` → `.tsx` 🟡 P3 (фон)
**Файлы:** 23 файла в `components/`.
**Почему последним:** чистый maintainability-долг, сборка зелёная — не баг. Без TD-1 ловить типам нечего.
**План:**
1. Сначала вынести/удалить `docs/prototypes/code/*.jsx` из дерева сборки (мёртвый код).
2. Переименовывать по файлу, начиная с компонентов с формами и данными тенанта (`QueueBoard`, `RadiologistBoard`, модалки бронирования/редактирования), презентационные — в конце.
3. На каждый файл: пропсы → интерфейсы, убрать `any`, опереться на `Database` и типы из TD-2/TD-4.
4. Метрика прогресса — счётчик оставшихся `.jsx` (цель: 0 в `components/`).
**Оценка:** 8–13 ЧД, итеративно (не блокирует другие задачи). **Зависит от:** TD-1.

## 5. Метрики успеха

- `supabase/types.ts` существует и регенерируется в CI после миграций; все клиенты с `<Database>`.
- `.js` в `lib/` = 0; `.jsx` в `components/` = 0.

- Мутации очереди/инцидентов идут через Server Actions с серверной валидацией; в досках — оптимистичные апдейты.
