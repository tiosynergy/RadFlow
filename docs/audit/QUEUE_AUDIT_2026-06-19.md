# RadFlow — Аудит логики формирования и ведения очереди

**Дата:** 2026-06-19 · **Область:** построение очереди, слоты, статусы, простои/поломки, перенос, CITO
**Метод:** трассировка кода по всем сценариям (QueueBoard, BookingModal, RescheduleModal, BreakdownModal, CallListBoard, lib/*, миграции БД).
**Файлы:** `components/QueueBoard.jsx`, `BookingModal.jsx`, `RescheduleModal.jsx`, `BreakdownModal.jsx`, `CallListBoard.jsx`, `lib/queueStatus.js`, `lib/schedule.js`, `lib/studies.js`, `supabase/migrations/0001–0017`.

## Вывод

Базовый happy-path (запись → пришёл → в кабинете → выполнено) и защита от двойной брони по времени (триггер `check_no_overlap`) — корректны и хорошо продуманы. Однако найдены **3 критических**, **3 средних** и **5 низких** логических дефекта. Главная системная проблема: **бизнес-инварианты очереди держатся только на клиентских проверках в одной кнопке, а альтернативные пути (кружки прогресс-степпера, гонки, второй инцидент на кабинет) их обходят.** Ниже — детали и пошаговый план.

## Критические

### C1. Кружки прогресс-степпера обходят все проверки вызова
**Где:** `QueueBoard.jsx` — `QueueRow`, `onClick={act(() => onSetStatus(p, key))}` (строки ~403) → `onSetStatus = (pt, st) => setStatus(pt.id, st)` (строка 1105) → `setStatus` (797).
**Суть:** Главная кнопка «Викликати в кабінет» идёт через `callPatient()` (885), который проверяет: кабинет не заблокирован простоем, не закрыт по графику, не занят другим `in_progress`. Но **клик по кружку шага «В кабінеті» вызывает `setStatus` напрямую**, минуя `callPatient`. Итог: можно перевести пациента в `in_progress`:
- в кабинет, заблокированный поломкой/ТО;
- в кабинет, закрытый по графику дня;
- когда там уже есть пациент `in_progress` → второй «в кабинете» одновременно.

**Последствие:** рассинхрон доски и реальности, «потерянный» пациент (см. C2).

### C2. Нет инварианта «один `in_progress` на кабинет»
**Где:** нет проверки ни в БД (миграции 0014/0016 ограничивают только пересечение по `scheduled_at`), ни устойчиво на клиенте. `currentByRoom[e.room_id] = e` (937–939) при двух `in_progress` **молча оставит последний**, первый исчезнет из карточки кабинета и из `CurrentCard`, но останется `in_progress` в БД.
**Последствие:** пациент «зависает» в кабинете невидимым; счётчики и загрузка врут; завершить его можно только из таблицы.
**Связано с:** C1 (главный способ создать ситуацию), гонка двух регистраторов (проверка `busy` в `callPatient` — клиентская, без блокировки).

### C3. CITO (срочные) не влияет на очередь
**Где:** `cito` используется только для бейджа (335) и баннера (765, 984). Сортировка очереди (`sorted`, 946–950) идёт по `FLOW[status]` затем по времени; `nextWaitingByRoom` (940–942) берёт первого `waiting` по времени. **Признак срочности нигде не поднимает пациента в очереди и не предлагается к вызову первым.**
**Последствие:** для продукта «управление очередью» срочный пациент не приоритизируется — ключевая функция отсутствует, хотя в UI заявлена.

## Средние

### M1. Два одновременных простоя на кабинет — «affected» видит только один
**Где:** `QueueBoard.jsx` 743–744: `incidents.forEach((i) => { incidentByRoom[i.room_id] = i; })` — **last-wins**. `BreakdownModal` явно позволяет на одном кабинете одновременно и «Поломку» (active), и «Планове ТО» (planned), а `loadIncidents` грузит оба. Детекция «постраждалих» (753–762) и `entryInIncidentWindow` смотрят только `incidentByRoom[room]` (один инцидент).
**Последствие:** пациенты в окне *второго* инцидента не попадают в панель «Обдзвін через простій» и не помечаются на перенос. При этом `BookingModal` (296–305) и `RescheduleModal` (63–73) проверяют **все** инциденты кабинета — налицо несогласованность: записать в окно нельзя, но уже записанных в это окно доска «не замечает».

### M2. Таймер «в кабінеті» считается от `updated_at` — сбрасывается на любом апдейте
**Где:** `enteredAt = currentByRoom[r.id].updated_at` (1043, 1065). `updated_at` обновляется триггером `touch_updated_at` (0001) при **любом** UPDATE строки: правка исследований, смена `call_status`, realtime-перезапись и т.п.
**Последствие:** время нахождения в кабинете и индикатор «перевищено час» врут (обнуляются), когда во время процедуры меняют что-либо в записи. Нужен отдельный момент входа `in_progress_at`.

### M3. Загрузка кабинетов считается от неверной ёмкости
**Где:** `computeRoomLoad` (251–258): `cap = 480` (8 ч). Реальный график по умолчанию 08:00–18:00 = **600 мин**; особый график/закрытые дни/простои не учитываются.
**Последствие:** проценты загрузки систематически завышены и недостоверны (полный день показывает >100%, обрезается до 100). Метрика вводит в заблуждение администратора и CEO-дашборд.

## Низкие / UX

### L1. «⚠ Уточнити» не срабатывает для `waiting`
`needsClarification` (lib/queueStatus.js) реагирует только на `scheduled`. Пациент, который «пришёл» (`waiting`), но не вызван, а время давно прошло — без сигнала. Логичнее подсвечивать и просроченный `waiting`.

### L2. Возврат в очередь может молча упасть в тупик
`undo` (807) для `not_held`/`no_show` ставит `scheduled`. Так как `not_held`/`no_show` освобождают слот (0016), слот мог быть переотдан → триггер отклонит UPDATE → только тост «Помилка». Нет подсказки «слот занят — перенесите». То же для `setCall`/любого возврата.

### L3. Фиксированная сетка слотов 30 мин не выравнивается по факт. окончаниям
`slotsList` шагает по 30 мин от начала графика. При нечётных длительностях (40/45/60 мин) образуются «дыры» (напр. 08:45–09:00), которые не предлагаются под запись — недоиспользование кабинета. Это дизайн-компромисс, но стоит зафиксировать как ограничение.

### L4. Все инварианты — клиентские; сервер защищает только пересечение по времени
Триггер `check_no_overlap` не знает про окна простоев (инцидентов) и про двойной `in_progress`. UI это закрывает, но при гонке/обходе UI/прямом запросе данные разойдутся. Защита «в глубину» отсутствует для двух из трёх инвариантов.

### L5. `CallListBoard.doReschedule` не нормализует статус
`CallListBoard.jsx` 263–276: перенос ставит `scheduled_at/date/time/duration` и `call_status:'confirmed'`, но **не выставляет** `status:'scheduled'` (в отличие от доски, `QueueBoard.doReschedule` 860–863). Для текущей выборки (только scheduled/waiting) не критично, но это скрытая несогласованность — при переносе `waiting`-пациента он останется `waiting` на будущей дате.

## Пошаговый план исправлений

Рекомендуемый порядок: C2/C1 → C3 → M1 → M2 → M3 → L*.

**Шаг 1. Закрыть инвариант «один `in_progress` на кабинет» (C1+C2).**
1. БД: добавить миграцию `0018_one_in_progress_per_room.sql` — частичный уникальный индекс:
   `create unique index ... on queue_entries(room_id) where status = 'in_progress';`
   (дедуплицировать существующие перед созданием, как в 0017).
2. Клиент: единственная точка перехода в `in_progress` — `callPatient`. В `QueueRow` для кружка `in_progress` и для шага вызывать `onCall` (с проверками), а не голый `onSetStatus`. Для остальных кружков (scheduled/waiting/done) оставить прямую установку, но прогонять через общий guard.
3. Обработать ошибку индекса в `setStatus`/`callPatient` человеко-понятным тостом («У кабінеті вже є пацієнт»).

**Шаг 2. Привести `onSetStatus` к проверкам (C1).**
Ввести `guardedSetStatus(p, status)`: для целевого `in_progress` — те же проверки, что в `callPatient` (блокировка, график, занятость); для `waiting`/`done` — без изменений. Степпер и кнопки используют только её.

**Шаг 3. Приоритет CITO (C3).**
1. В `sorted` добавить первичный ключ сортировки: `cito && активный статус` → выше.
2. `nextWaitingByRoom`: среди `waiting` выбирать сначала `cito`, затем по времени.
3. В баннере CITO кнопку «Показати чергу» дополнить «Викликати першочергово» (вызов первого активного CITO).

**Шаг 4. Учитывать все инциденты кабинета (M1).**
Заменить `incidentByRoom` (map одного) на `incidentsByRoom` (массив). В детекции «affected» и `entryInIncidentWindow` проверять `some(...)` по всем инцидентам комнаты — как уже сделано в BookingModal/RescheduleModal. Баннеры уже идут по `incidents.filter(...)` — согласовать.

**Шаг 5. Отдельная метка входа в кабинет (M2).**
1. Миграция: `add column in_progress_at timestamptz`.
2. При переходе в `in_progress` писать `in_progress_at = now()` (и обнулять при выходе/возврате).
3. Таймер (`LiveTimer`, `enteredAt`) читать `in_progress_at` вместо `updated_at`.

**Шаг 6. Корректная ёмкость загрузки (M3).**
В `computeRoomLoad` считать `cap` из эффективного графика кабинета на дату (`roomScheduleFor` → `end−start` в минутах), вычитать окна простоев; закрытый день → cap 0 / «Зачинено».

**Шаг 7. Низкие.**
- L1: `needsClarification` — добавить ветку для `waiting` с прошедшим временем (отдельный текст «прийшов, не викликаний»).
- L2: в `setStatus`/`undo`/`doReschedule` при `exclusion_violation` показывать тост «Слот уже зайнятий — перенесіть на інший час» и подсвечивать кнопку «Перенести».
- L3: задокументировать ограничение сетки; опционально — режим «впритул» (слоты от конца предыдущей процедуры).
- L5: в `CallListBoard.doReschedule` добавить `status: 'scheduled'`.

**Шаг 8. Регресс-проверка (обязательно).**
Прогнать сценарии: два регистратора вызывают в один кабинет; клик по кружку «В кабінеті» на заблокированном/закрытом/занятом кабинете; кабинет с поломкой + ТО одновременно (проверить панель «Обдзвін через простій»); срочный CITO — порядок вызова; правка исследований у пациента «в кабінеті» (таймер не должен сбрасываться); загрузка кабинета при полном дне (≤100% и реалистично); возврат `not_held` на переотданный слот (понятная ошибка).

## Что корректно (не требует правок)
- Триггер `check_no_overlap` + advisory-lock — надёжная защита от двойной брони по времени; `not_held`/`no_show`/`cancelled` корректно освобождают слот.
- Закрытие «жёсткого 18:00»: слот, не вмещающийся до конца графика, помечается `tight`.
- Слоты строятся по эффективному графику кабинета (`roomScheduleFor`) с учётом override.
- Планируемое будущее ТО (`planned`) не блокирует кабинет заранее (`blockingByRoom` по текущему времени).
- `один активный інцидент на кабінет` (0017) для статуса active.
