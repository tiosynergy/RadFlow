CITO Priority

🚨 CITO is the only priority flag (the "Urgent" status was removed in v3.1 for consistency). A CITO patient triggers a red banner on the Admin Queue Board and in the Radiologist Office simultaneously. CITO patients always appear at the top of their status group.

Schedule Generation Logic (schedule.js)

🔢 getDayPatients(date) uses a deterministic seeded PRNG (mulberry32) keyed by date. The same day always returns the same schedule — but different days return different queues. Sunday = closed, Saturday = reduced, weekday = full. Past days show completed statuses; future days show queued.

Incident Management — Поломка / ТО (v3.6)

🔧 Реєстрація поломки/ТО апарата автоматично формує колл-лист на перезапис і блокує кабінет. Кілька кабінетів можуть мати інциденти одночасно — rf_incident_v1 зберігає масив, по одному інциденту на кабінет.
КрокЩо відбувається
ДіалогАдмін обирає апарат + причину (поломка / планове ТО) + початок + тривалість. Тривалість «До відновлення» додає поле дати відновлення.
ПостраждаліrfBuildIncident() → rfIncidentAffected(): усі незавершені записи апарата сьогодні + (для «До відновлення») майбутні записи по днях до дати відновлення (getDayPatients). Кожен запис має dayLabel/isToday.
ЗбереженняupsertIncident() пише інцидент у масив (за roomKey) → подія rf-incident-sync.
ВідображенняДошка черги: банер + картка «🔒 Заблоковано» + панель «Обдзвін через простій» (на кожен інцидент) + позначки «🔧 Перезапис» у черзі. Сторінка «Інцидент»: активні блоки. Колл-лист + сайдбар: сумарні лічильники.
ОбдзвінОператор фіксує статус кожного пацієнта: Перезаписано / Передзвонити / Відмова (setIncidentCallStatus). Лічильник «N на обдзвін» зменшується.
Редагування«✏ Редагувати» перезбирає інцидент; rfMergeIncidentStatuses() зберігає вже зафіксовані статуси обдзвону.
РозблокуванняremoveIncident(roomKey) — точково на кабінет; інші інциденти лишаються.
🎯 Закриває евристики Нільсена: видимість статусу системи (банер + лічильник), відповідність реальному світу (термінологія простою/обдзвону), запобігання помилкам і контроль користувача (діалог з апаратом/тривалістю замість «німого» кліку).

Flow Simulator (flow-sim.js)

▶ Automatically advances patients through the queue: Queued → Waiting → In Cabinet → Done (10% chance of No-show). Steps every 12s (demo speed = 3× slower than original). A Lead-lock via localStorage ensures only one tab drives the simulation at a time — others are passive consumers. Toggle: "Flow: Live / Pause" in the Admin header.
